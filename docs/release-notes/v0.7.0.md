# v0.7.0

## Theme-aware markdown rendering

- Fixed unreadable markdown in dark themes: MdXaml's built-in GithubLike style had hardcoded light-theme colors for headings (black), table rows (white/near-white), code blocks (nearly transparent), blockquote borders (light gray), and note boxes (white background)
- Created shared `BaseMarkdownStyle` in `Themes/MarkdownStyles.xaml` that overrides all GithubLike sub-element styles with `DynamicResource` lookups, covering: headings (H1-H4), code blocks, inline code spans, tables (borders, header rows, alternating rows), blockquotes, notes, and hyperlinks
- Added 9 new theme color keys (`MarkdownHeadingForeground`, `MarkdownCodeBackground`, `MarkdownTableBorderBrush`, `MarkdownTableHeaderBackground`, `MarkdownTableEvenRowBackground`, `MarkdownBlockquoteBorderBrush`, `MarkdownNoteBackground`, `MarkdownNoteBorderBrush`, `MarkdownLinkForeground`) to all 6 themes with appropriate colors for each
- Both AI chat panel and file preview pane now derive from the shared base style, ensuring consistent theme-aware markdown across the app
- Code blocks now use Cascadia Code font family (consistent with rest of the app) instead of Courier New

## File preview close button

- Added an X button to the file preview pane that clears the current preview and returns to the empty state
- Button appears in the top-right corner whenever a file, diff, or image is being previewed

## Fix bold/italic markdown links

- Fixed bold and italic text showing raw asterisks when wrapping links (e.g. `**[text](url)**`)
- MdXaml parses links before bold/italic, so `**[link](url)**` was never recognized as bold; added a pre-processing step that rewrites these patterns to `[**link**](url)` before rendering

## Fix taskbar activation with custom chrome

- Fixed a bug where clicking the taskbar icon would produce a "ding" instead of bringing the window to the front
- Caused by `WindowStyle=None` stripping the `WS_SYSMENU` Win32 style bit, which prevents the shell from properly activating the window; re-added via `SetWindowLong` in `OnSourceInitialized`
- Fixed `LaunchTool` (VS Code, Cursor) using `UseShellExecute=false` with `cmd.exe`, which created parent-child process relationships that could steal foreground activation rights; now uses `UseShellExecute=true` to launch fully independent processes
